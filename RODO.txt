GUI: TKINTER → PYSIDE6
-----------------------
Arrancó en Tkinter (sin dependencias externas). Se cambió a PySide6/Qt
porque se ve mucho más nativo/prolijo de entrada, y porque su hoja de
estilos (QSS) es casi CSS, lo que calza mejor con el design system que
Tkinter/ttk. El cambio quedó contenido en la capa `ui.py` de cada módulo
más `src/app.py` y `main.py` — `db.py`/`logic.py` no se tocaron, porque ya
estaban separados de la interfaz.

Costo real del cambio: deja de ser "sin dependencias externas", ahora
`requirements.txt` pide `PySide6` (LGPL, gratis incluso para distribuir a
otros locales — a diferencia de PyQt6, que es GPL/comercial).

Convenciones nuevas por el cambio de framework:
- Pantallas como QWidget, layouts con QVBoxLayout/QHBoxLayout (no más
  pack/place).
- Botones con `objectName` (primaryButton, secondaryButton) en vez de
  `style=` de ttk, para que el QSS los tipee por selector de id.
- MainWindow con QStackedWidget para alternar entre pantallas (login /
  menú principal), reemplaza el manejo manual de frames de Tkinter.


ESTILOS
--------
No se define paleta propia: se usa el design system interno (fuente de
verdad, no se clona, solo se copia/adapta el archivo que corresponda).
https://github.com/EnzoAlesandro17/design-system

Ahí todavía no hay una versión para PySide, así que `src/constants/
styles.py` arma el QSS a mano (función `build_stylesheet`), pero con los
mismos valores hex que el resto del design system. Uso:

    from src.constants.styles import build_stylesheet

    ventana.setStyleSheet(build_stylesheet(dark=False))

Cualquier ajuste de color se decide primero en el repo del design system,
no acá — este archivo solo lo traduce a QSS.

La barra superior es la excepción: siempre usa el azul oscuro del modo
claro (`HEADER_COLOR`, fijo), no cambia con el toggle de tema.


MODELOS DE DATOS — branches / employees / users
--------------------------------------------------
Los tres módulos base ya tienen CRUD completo (listar, crear, editar,
borrado lógico) con validaciones mínimas (campos obligatorios, formato de
email, código único).

- branches: id, code (único), name, address, street_number, email, city,
  state, country, status.
- employees: id, code (único), name, last_name, phone, email, branch_id
  (FK a branches), status.
- users: id, username (único), password (hasheada, pbkdf2_hmac), email,
  role, employee_id (FK a employees, opcional), status.

Dos cosas que sumé sobre lo pedido, para no perder decisiones ya tomadas
antes en este mismo RODO:
- `role` en users: no estaba en la lista de campos, pero sin esto se
  pierde la distinción IT/Supervisor/Encargado/Vendedor ya definida más
  arriba. Lista de roles centralizada en `Settings.ROLES`.
- `status` en las tres tablas: borrado lógico, ya decidido en
  DECISIONES DE ARQUITECTURA de la primera etapa.

Pendiente, no bloqueante: permissions.py sigue vacío — con 3 módulos ya
andando es buen momento para definir permiso fino por rol y acción.


ABSTRACCIÓN: CrudListWidget
------------------------------
Sucursales/Empleados/Usuarios repetían la misma pantalla de listado
(tabla + Agregar/Editar/Eliminar/Volver), solo cambiando columnas y el
diálogo de formulario. Eso sí se extrajo a `src/widgets/
crud_list_widget.py` (`CrudListWidget`), porque era mecánico de verdad —
no había reglas de negocio escondidas en la duplicación.

Cada módulo hereda y define: `titulo`, `columnas`, `listar()`,
`fila_a_valores(fila)`, `abrir_dialogo(id_=None)` (devuelve el QDialog de
alta/edición) y `eliminar(id_)`. El QDialog de cada módulo (con sus
campos/combos propios) sigue siendo especifico — ahí sí hay diferencias
reales entre módulos (validaciones, combos de FK, hash de contraseña en
Usuarios) como para no forzar una abstracción.

Agregar/Editar/Eliminar quedaron los tres abajo, con el mismo estilo
(`secondaryButton`, el mismo que ya usaban Editar/Eliminar) — no hizo
falta agregar nada nuevo a `styles.py` para esto.
