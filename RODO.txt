RODO - notas personales sobre AppEmar
=====================================
(esto es un readme mío, para mí. No es documentación oficial del proyecto)


QUÉ HICIMOS
-----------
- Migramos el main.py viejo (Tkinter + Excel/xlwings, la "CAJA") a una
  arquitectura modular: cada área (users, clients, products, branches) tiene
  su propio db.py / logic.py / ui.py, todos compartiendo la misma DB SQLite.
- Conexión centralizada en src/db/connection.py (GestorDB.conectar() +
  obtener_conexion() como context manager, con foreign_keys ON).
- Esquema de las 4 tablas, todas con columna status (1 = activo, 0 = borrado
  lógico) para no perder datos con un DELETE real.
- CRUD completo + validaciones de negocio en los 4 módulos, con las
  validaciones genéricas (email, teléfono, DNI, CUIT, fecha/edad mínima,
  campos obligatorios) centralizadas en src/constants/validations.py para
  no duplicarlas módulo por módulo.
- Contraseñas de usuario hasheadas con pbkdf2_hmac + salt (nunca texto plano).
- products: precio mayorista/minorista con validación cruzada, y stock que
  puede ser NULL para productos sin inventario físico (ej: E-SIM).
- main.py ahora crea las tablas solo al arrancar (inicializar_db) y muestra
  un sidebar con los 4 módulos (por ahora con UI placeholder, "en
  construcción").
- De paso resolvimos cosas del entorno: tkinter roto en el sandbox de VS Code
  Flatpak, tamaño de ventana (900x500), requirements.txt corrupto en UTF-16
  con dependencias viejas de Excel, y código muerto en config.py.
- (Update) El encoding de requirements.txt en realidad seguía en UTF-16LE
  pese a lo anotado arriba (rompía pip install -r requirements.txt con
  UnicodeDecodeError) — ahora sí quedó en UTF-8 de verdad, verificado
  corriendo pip install.
- users gana columna role (Admin / Gerente / Vendedor / Invitado),
  validada en crear_/actualizar_usuario. Por ahora el rol se guarda y se
  muestra, no restringe todavía qué módulos ve cada uno.
- Login real: pantalla LoginFrame (src/modules/user/login_ui.py) usando
  iniciar_sesion() (nueva, con mensaje de error genérico para no filtrar
  si falló el usuario o la contraseña), conectada por primera vez a un
  messagebox de error — hasta ahora ninguna pantalla atrapaba
  ValidationError. Se bloquea el formulario a los MAX_LOGIN_ATTEMPTS (3)
  intentos fallidos, constante que ya estaba definida y sin usar.
  main.py arranca en el login en vez del sidebar directo.
- src/session.py: módulo chico que guarda el usuario logueado en memoria
  para que cualquier pantalla futura lo pueda consultar sin pasarlo a
  mano por todos los frames.
- Primera UI de CRUD real del proyecto: el módulo Usuarios dejó de ser un
  placeholder. Treeview con la lista, y diálogos separados para
  alta/edición (UsuarioFormDialog, con combobox de rol) y cambio de
  contraseña (CambiarContrasenaDialog). Va a servir de molde para cuando
  se hagan las UI de clients/products/branches.
- src/permissions.py: primer lugar centralizado para reglas de "qué rol
  puede hacer qué" (puede_gestionar_usuarios, puede_cambiar_contrasenas),
  mismo espíritu que validations.py — se arma la primera vez que hace
  falta, no cuando ya se repitió en varios módulos.
- Reglas de permisos de Usuarios acordadas: crear/eliminar/editar datos
  solo Admin y Gerente; cambiar contraseña (de cualquier usuario) también
  lo puede hacer un Vendedor; Invitado ve la lista en solo lectura sin
  ningún botón.


QUÉ SEGUIRÍA HACIENDO
----------------------
- Las UI reales de clients/products/branches (hoy siguen siendo un Label "en
  construcción") — calcar el patrón de Usuarios (Treeview + diálogos
  separados de alta/edición) que ya quedó armado y probado.
- Reactivar usuarios borrados desde la UI (reactivar_usuario ya existe en
  logic.py, pero todavía no hay botón que lo use).
- Si en algún momento hace falta, restringir acceso a módulos por role (hoy
  el rol se guarda y se muestra pero no bloquea nada del sidebar).
- Un módulo de ventas/movimientos que relacione clients + products + users +
  branches. Hoy las 4 tablas son independientes entre sí, sin foreign keys
  reales unas con otras (branch_id, client_id, etc.) — eso va a hacer falta
  en cuanto haya que registrar una venta.
- Tests automatizados (pytest) en vez de los scripts sueltos que fui corriendo
  a mano en cada paso para probar cada módulo. Esos tests no quedaron guardados
  en ningún lado del repo.
- Migraciones de esquema reales (ALTER TABLE) para cuando ya haya datos
  cargados en producción y haya que sumar columnas sin perder información
  (hoy resolvemos esto borrando la DB de desarrollo, que no sirve una vez
  que el sistema esté en uso real).
- Empaquetar la app para Windows (PyInstaller o similar), ya que el uso final
  es en la compu del local.
- Conectar ValidationError con messagebox en el resto de las pantallas (el
  login ya lo hace, pero las UI de clients/products/branches/users siguen
  siendo placeholders sin ningún manejo de errores todavía).


CONSEJOS (para vos, che)
------------------------
- Antes de que el proyecto crezca más, probá escribir de cero un logic.py
  nuevo (o un módulo chico) replicando el patrón ya armado, sin ayuda. Vas a
  fijar mejor los conceptos que si solo los leés.
- Cuando conectes la UI real, probá primero a mano (como hicimos acá con
  scripts) antes de cablearlo todo a botones — así separás "¿la lógica anda?"
  de "¿la interfaz anda?".
- Guardate en algún lado (acá mismo o en la cabeza) las decisiones de diseño
  que fuimos tomando juntos: por qué status en vez de DELETE, por qué DNI/CUIT
  autodetectado, por qué stock NULL para E-SIM. Si volvés a este código en 3
  meses te vas a agradecer el contexto.
- No tengas miedo de pedir que algo se centralice/refactorice apenas notes
  que se repite (como hiciste con las validaciones) — es mucho más barato
  hacerlo ahora, con 4 módulos, que dentro de un año con 10.


COSAS BUENAS
------------
- La decisión de separar en db/logic/ui desde el arranque fue sólida — hizo
  que cada módulo nuevo (branches, en particular) fuera calcar el patrón
  anterior en vez de inventar de nuevo.
- Fuiste vos quien pidió centralizar las validaciones compartidas en vez de
  dejarlas duplicadas — ese es exactamente el tipo de intervención que
  mantiene el código escalable a medida que se agregan módulos.
- Pensaste en casos borde reales del negocio antes de que se conviertan en
  bugs (E-SIM sin stock, DNI vs CUIT, precio mayorista/minorista) en vez de
  descubrirlos después con datos reales cargados.


COSAS A MEJORAR
----------------
- Cero tests automatizados por ahora. Todo lo que probamos fue a mano, en la
  conversación — nada de eso se vuelve a ejecutar solo si cambiás algo.
- Las tablas viven aisladas unas de otras (sin FK reales entre ellas todavía).
- El manejo de errores en pantalla (messagebox) ya cubre login y Usuarios;
  falta todavía en clients/products/branches, que siguen siendo placeholders.
- README.md sigue vacío — en algún momento conviene, aunque sea 4 líneas,
  para cuando otra persona (o vos en 6 meses) abra el repo de cero.


ALGO EXTRA
----------
Ya que es para un local Claro: capaz en algún momento el sistema crezca más
allá de "clientes/productos/usuarios" — por ejemplo, si en algún momento
querés trackear altas de línea o portabilidades, no solo venta de equipos.
No hace falta armar nada de esto ahora, pero si lo tenés en mente desde ya
te va a evitar tener que rehacer el esquema de las tablas de cero el día que
lo necesites.
